## 1. PURPOSE

This document establishes the master framework for how Prisca Dezigns handles all data processed, generated, transformed, or stored by AI systems. It governs every data touchpoint — from the moment a client submits information, through AI processing, to final output delivery and archival.

Every specialized AI Data SOP (SOP-AD-001 through SOP-AD-003) derives its rules from this master framework.

---

## 2. SCOPE

This SOP applies to:
- All AI agents operating within or on behalf of Prisca Dezigns (Drew, Sierra, Kimi, and any future agents)
- All client data received via any channel (WhatsApp, email, web forms, file uploads, API)
- All third-party platforms used in data workflows (Supabase, Google Drive, Make.com, n8n, GitHub)
- All team members, contractors, and sub-processors with data access
- All sub-brands and client projects managed through the agency

## 3. DEFINITIONS

| Term | Definition |

| **Data Subject** | The individual whose data is being processed |
| **Retention Period** | The defined length of time data is kept before secure deletion |

## 4. CORE DATA PRINCIPLES (ISO 8000 + GDPR Art. 5)

All AI data operations at Prisca Dezigns must satisfy all six principles:

| Principle | Requirement |
|---|---|
| **Lawfulness** | Data must be collected with a valid legal basis (consent, contract, legitimate interest) |
| **Purpose Limitation** | Data collected for one purpose may not be reused for a different purpose without re-consent |
| **Data Minimisation** | Only the minimum data necessary for the task may be collected or processed |
| **Accuracy** | AI outputs must be verified for accuracy before delivery; minimum threshold: 98% |
| **Storage Limitation** | Data must not be retained beyond the defined retention period for each category |
| **Integrity & Confidentiality** | All data in transit and at rest must be encrypted; access must be role-based |

## 6. DATA LIFECYCLE MANAGEMENT

### 6.1 Ingestion
1. All raw input data is received and logged with a timestamp and source identifier
2. Data is classified (Class 1–4) at the point of ingestion
3. Class 3 and Class 4 data is flagged for encryption before storage
4. A data inventory entry is created in Supabase (`data_log` table)

### 6.2 Processing
1. AI models may only access data required for the specific task (Data Minimisation)
2. No AI model may store raw PII in its context beyond the active session
3. All AI processing must be logged with: input hash, model used, output hash, timestamp
4. Outputs must be validated before delivery (accuracy threshold: 98% for structured data, 90% for AI-generated content)

### 6.3 Storage
1. All structured data is stored in Supabase with row-level security enabled
2. Files and documents are stored in Google Drive with restricted sharing settings
3. Encryption at rest: AES-256 (Supabase default)
4. Encryption in transit: TLS 1.2 minimum for all API calls

### 6.4 Access Control
1. Role-based access — no team member or AI agent may access data beyond their role scope
2. All API keys and credentials must be stored in credential vaults — never hardcoded
3. Access logs must be retained for a minimum of 12 months
4. Quarterly access review required — revoke all inactive access

### 6.5 Retention Schedule

| Data Category | Retention Period | Action at Expiry |
|---|---|---|
| Client PII (active contract) | Duration of contract + 1 year | Anonymise or delete |
| Client PII (prospect/lead) | 90 days from last contact | Delete |
| AI processing logs | 12 months | Archive then delete |
| Financial records | 7 years (TT Companies Act) | Secure archive |
| SOP documents | Indefinite (version-controlled) | Archive superseded versions |
| Consent records | Duration of consent + 3 years | Delete after expiry |

### 6.6 Deletion
1. Upon contract end or client request, all Class 3 and Class 4 data must be deleted within 30 days
2. Deletion must be verified and logged with a confirmation record
3. Supabase: row deletion + vacuum; Drive: permanent delete from trash
4. A Deletion Certificate is issued to the client upon request

---

## 7. CONSENT MANAGEMENT

1. All data collection must be preceded by informed consent in plain language
2. Consent must be specific — one consent per purpose
3. Opt-out must be as easy as opt-in — one-click or one-message
4. Consent records are stored in Supabase with: timestamp, channel, consent text hash, data subject identifier
5. Consent withdrawal is processed within 48 hours
6. AI agents must never process data from a subject who has withdrawn consent

## 8. DATA BREACH PROTOCOL

| Stage | Action | Timeline |
|---|---|---|
| **Detection** | Identify and isolate affected system | Immediate (within 1 hour) |
| **Assessment** | Determine scope — what data, how many subjects | Within 4 hours |
| **Containment** | Revoke compromised credentials, patch vulnerability | Within 8 hours |
| **Internal Notification** | Notify Lead Strategist | Within 2 hours of detection |
| **Client Notification** | Notify affected clients with incident report | Within 48 hours |
| **Regulatory Notification** | Notify TT Information Commissioner (if required) | Within a reasonable time (TT DPA 2011); 72 hours (GDPR if applicable) |
| **Post-Incident Review** | Root cause analysis, SOP update if needed | Within 14 days |
